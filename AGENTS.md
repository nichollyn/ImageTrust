# Claude Code 协同开发最佳实践

## 概述

本目录用于与 Claude Code 协同开发项目。每次启动新项目时，参考本文件进行初始化设置。

**顶级规则**：对本文档的任何说明、修改或解释，**必须经过人工确认**后才能执行。

**优先级原则**：本文档中描述的规则在上下文中始终具有较高优先级，应优先遵循。

---

## 需求守门员

### 核心理念

执行力不能替代判断力。在动手实现之前，始终以审视者的视角检视需求本身。

### 触发条件

当用户提出新需求或功能请求时，在进入实现之前，评估以下三个问题：

1. **目的**：这个需求服务于项目的什么核心目标？
2. **必要性**：它是否真的需要现在实现？还是可以推迟、简化、甚至不做？
3. **边界**：实现到何种程度是合适的？最小可行方案是什么？

### 项目目标锚定

每个项目应在 CLAUDE.md 或 WORKLOG.md 中明确记录其**核心定位**（一句话描述项目存在的意义）。当新需求偏离这个定位时，主动与用户确认：

> "这个需求看起来偏离了项目最初的核心目标 [X]。是你的方向有调整，还是我们先聚焦在 [X] 上？"

### 不做什么

- 不因为"能做"就去做——能力不等于必要性
- 不在用户的兴奋中失去独立判断——用户可能因为 AI 执行力强而不自觉地扩大范围
- 不把"用户说了"当作唯一的行动依据——理解意图比执行指令更重要

### 怎么做

- 对于明显偏离核心目标的需求：**暂停并确认**
- 对于合理但范围偏大的需求：**建议最小可行方案**，而非一次性做到位
- 对于与核心目标一致的需求：**直接执行**，不增加不必要的确认摩擦

## Slash 命令学习

### 核心理念

借助"做中学"(learn by use, learn when need) 的方式，在实际交互场景中掌握 `/` 命令的使用。

### 触发提醒的场景

当多次自然语言对话出现以下信号时，应主动提醒用户使用 `/` 命令：

- **帮助理解**：需要我更快速理解用户的意图
- **避免 side question**：冗长的多轮对话可以用单个命令替代
- **提升效率**：明显可以通过预设命令一次性完成的操作

### 评价标准

"更优"指：

1. 减少对话轮次，节省交互时间
2. 更快达成共识或完成操作
3. 避免重复性说明

### 学习策略

- 不强制记忆所有命令，在实际场景中学习
- 当遇到适合用命令的场景时，提醒并解释
- 逐步积累，达成"需要时即懂"的自然状态

## 新项目初始化检查清单

### 1. Git 仓库初始化

每个项目必须初始化 git 仓库，并在完成后做初始提交：

```bash
git init
git add -A
git commit -m "Initial commit: <项目名称和描述>"
```

**重要**：在完成重要修复或功能时及时提交，并写清楚"为什么这样改"。这样当上下文丢失时，可以从 git 历史恢复。

### 2. 项目级文件结构

每个项目应包含以下文件：

| 文件 | 用途 |
|------|------|
| `CLAUDE.md` | 项目级规则和方法论 |
| `WORKLOG.md` | 当前任务状态、进度、问题记录 |
| `.claude/memory/MEMORY.md` | 项目记忆、核心组件、已知问题 |

详细格式和示例请参考 [CLAUDE_EXAMPLES.md](./CLAUDE_EXAMPLES.md)。

### 3. SessionEnd Hook 配置

推荐配置会话结束时的自动备份机制。

详细配置脚本和 settings.json 示例请参考 [CLAUDE_EXAMPLES.md](./CLAUDE_EXAMPLES.md#sessionend-hook-配置)。

### 4. .gitignore

为项目类型配置适当的忽略规则。

**必须忽略的系统文件和目录**：

- `.DS_Store` - macOS 系统文件

各技术栈的 .gitignore 示例请参考 [CLAUDE_EXAMPLES.md](./CLAUDE_EXAMPLES.md#gitignore-示例)。

### 5. 平台/技术栈特定规则

当确定目标平台和技术栈后，应加载对应的规则文件。

**查找规则**：在顶级规则目录（`vcoding/`）中遍历 `{Name}.md` 文件，选择匹配的平台或技术栈规则文件加载，如 `ApplePlatform.md`、`Flutter.md`。

**兜底机制**：如果找不到对应文件，继续执行不报错。

## 上下文恢复

当上下文丢失后，用户说"resume"时：
1. 读取项目的 WORKLOG.md 了解当前状态
2. 读取 CLAUDE.md 确认项目上下文
3. 使用 `git log` 查看历史和改动原因
4. 使用 `git diff <commit>` 查看具体改动

## 存档触发词

以下词汇是明确的存档指令：

| 你说 | 我执行 |
|------|--------|
| `"先走了"` | git commit + 更新 WORKLOG（会话中断 checkpoint） |
| `"resume"` | 读取 WORKLOG + git log，重建上下文 |
| `"更新 WORKLOG"` | 将当前状态写入 WORKLOG |
| `"帮我提交"` | git commit 当前更改 |
| `"加载顶级规则"` | 从当前工作目录的外层目录读取最新的 CLAUDE.md，更新上下文。更新完成后，简要说明此次更新的新发现。特别地，这个目录常取名为 vcoding |
| `"同步顶级规则"` | `cp CLAUDE.md AGENTS.md` + git commit，确保 Kimi Code 和 Claude Code 读取的顶级规则内容一致 |

### 工具触发词

以下是对特定工具的口语化触发词，说这些词时 agent 应直接调用对应工具，不拆解为手动步骤：

| 你说 | 工具/动作 |
|------|----------|
| **"执行 xclingual"** | 调用 XCLingual MCP `auto_translate`，走完整本地化流程（lint → 补 comment → 提取 → AI 翻译 → 写入 → 验证 → 沉淀） |

**规则**：所有触发词的**唯一真源定义**位于本文档（`CLAUDE.md`）。子项目文档（`AGENTS.md`、`docs/` 等）只准引用，不准重复定义，避免多 agent 实例之间出现信息漂移。

**使用方式**：
- 不需要刻意描述范式，直接说意图
- 如果需要存档，说"更新 WORKLOG"然后描述内容，我会格式化写入
- 离开时说"先走了"，我自动执行 checkpoint
- git仓库如果存在追踪中的远程分支，commit后推送到对应的远程分支

## 会话结束前

在结束会话前：
1. 确保所有重要更改已提交到 git
2. 更新 WORKLOG.md 记录当前状态
3. 更新 MEMORY.md 记录关键信息

---

## 最佳实践原则

1. **每次重要修复后立即提交 git** - 不要等到累积很多更改后才提交
2. **提交信息要说明"为什么"** - 而不只是"做了什么"
3. **WORKLOG.md 要保持最新** - 让下一个会话能快速了解状态
4. **代码改动要简洁** - 避免引入不相关的修改
5. **子 Agent 模型选择** - 衍生子 Agent 时应合理评估子任务的复杂度，选择匹配的模型（opus/sonnet/haiku），避免简单任务使用高成本模型造成 token 浪费。每当完成了需要衍生子 Agent（拆分子任务）的任务时，汇报各子任务选用模型的情况

## 自动迭代模式

### 核心理念

自动迭代模式的目标是：在确保目标不偏移的前提下，实现连续数小时无需用户干预的自动化开发。

### 关键挑战

1. **视觉验证的局限性**：图像分析无法保证100%准确，应优先使用数值化验证
2. **调试日志的必要性**：静默失败会丢失关键信息，应使用明确的错误处理
3. **确认机制的前置**：需要用户确认的步骤应在迭代开始前一次性确认

### 验证机制原则

| 验证方式 | 适用场景 | 可靠性 |
|---------|---------|--------|
| 数值化输出（坐标、尺寸等） | 坐标计算、布局验证 | 高 |
| 文件写入确认 | 调试日志、导出文件 | 高 |
| 截图保存 + 人工确认 | UI布局、视觉呈现 | 中（需用户确认） |

**原则**：优先使用数值化验证，其次是文件确认，最后才考虑截图分析。

### 调试日志规范

**文件写入优先**：调试日志应优先写入文件系统，而非通过控制台或界面输出。这样 AI 可以自行读取结果，无需用户转达，提高调试效率。

**操作流程**：
1. 规划日志目录和文件名（应便于识别和后续查找）
2. 向用户说明写入计划（目录、文件名、日志内容概要）
3. 获得确认后执行写入
4. 写入完成后自行读取验证
5. 同步与用户对问题的理解，确保双方对日志内容的认知一致

**原则**：
- 使用明确的错误处理（`try-catch` / `do-catch`），避免静默失败
- 记录关键变量的值，便于问题追溯
- 日志内容应包含足够的上下文信息

- 示例代码请参考 [CLAUDE_EXAMPLES.md](./CLAUDE_EXAMPLES.md#调试日志规范)

### 自验证机制

在关键函数中内置"正确性断言"：
- 坐标转换后检查边界
- 输入参数验证
- 输出结果合理性检查

示例代码请参考 [CLAUDE_EXAMPLES.md](./CLAUDE_EXAMPLES.md#代码示例)。

### 任务分解与前置确认

复杂任务应分解为：
1. **子步骤清单**：列出所有需要执行的步骤
2. **条件分支**：每个步骤的"如果...则..."逻辑
3. **终止条件**：何时应该停止并报告

详细格式示例请参考 [CLAUDE_EXAMPLES.md](./CLAUDE_EXAMPLES.md#自动迭代计划示例)。

### 迭代过程透明化

每个迭代步骤都应输出可追溯的日志，包含：
- **操作**：具体做了什么
- **预期**：期望的结果
- **实际**：实际观察到的结果
- **结论**：基于差异的分析

### 自动迭代与人工干预的平衡

| 场景 | 建议行为 |
|------|---------|
| 明确的问题 + 确定的修复方案 | 自动执行，事后报告 |
| 不确定的问题 | 停止并请求确认 |
| 异常/错误 | 停止并详细报告 |
| 需要用户验证的结果 | 暂停等待确认 |

### 配置文件位置

自动迭代相关的项目特定配置可记录在：
- `WORKLOG.md` - 当前迭代状态和计划
- `.claude/memory/MEMORY.md` - 项目记忆和已知问题

## AI 知识库

基于协作经验积累的知识库，用于提升同类项目的协作效率。

### 核心理念

- **经验沉淀**：将解决过的问题和最佳实践记录下来
- **同类复用**：在类似项目中快速应用已有经验
- **持续迭代**：知识库随协作深入不断完善

### 组织结构

知识库位于 `AIKnowledge/` 目录，为单独git仓库管理，按技术栈、应用类型等维度划分粒度：

- 每个独立主题或领域一个文件
- 文件过大时应考虑拆分
- 避免过于琐碎的碎片化

### 内容来源

- 解决过的问题及解决方案
- 最佳实践和经验总结
- 常见错误及避免方法
- 技术选型考量

### 使用方式

在项目协作中，可根据需要加载 `AIKnowledge/` 下相关知识文件到上下文。

### 主题选择标准

适合加入知识库的主题应满足：

- **足够的通用性**：同类问题在不同项目中都可能遇到
- 具备复现或参考价值
- 非特定项目的专有内容

### 知识库操作指令

| 你说 | 我执行 |
|------|--------|
| `"备选知识库"` | 针对近期处理的问题或主题，选出有助于后续同类问题快速解决的几个，总结成文档并请求写入权限 |
| `"写入知识库"` | 将确认的选项写入 AIKnowledge 知识库 |
| `"更新知识库"` | 总结最近项目推进中新学到的知识和经验，写入 AIKnowledge 知识库（判断写入已有文件或创建新文件，不同主题可分别处理），完成后提交 AIKnowledge git 仓库 |
| `"同步知识库到skill"` | 读取 AIKnowledge 源文件内容，将其内嵌到 ~/.claude/skills/ 下对应的 SKILL.md 中（从薄壳模式切换为内嵌模式） |

### 知识候选池（自动沉淀机制）

**目标**：降低知识沉淀的人工触发依赖，通过"随手记 + 阈值触发"实现半自动化。

**机制**：

1. **随手记**：每次会话中发现"可能值得沉淀"的模式时，在项目的 `.claude/memory/MEMORY.md` 中的 `## 知识候选池` 章节追加一行，格式：`- [主题标签] 一句话描述 (出现次数: N)`
2. **阈值触发**：当同一主题标签出现 ≥ 2 次时，在下次会话中主动建议写入知识库（而非等用户说"更新知识库"）
3. **批量处理**：用户说"更新知识库"时，除写入确认内容外，还扫描候选池中已达阈值的项一并处理
4. **清理**：写入知识库后的条目从候选池中移除

**什么算"值得候选"**：
- 解决了一个非显而易见的问题（不是一搜就有答案的）
- 发现了一个反模式或陷阱
- 总结出一个可复用的代码模式
- 纠正了一个常见误解

**什么不算**：
- 特定项目的业务逻辑
- 一次性的 debug 过程（除非揭示了通用陷阱）
- 已在知识库中存在的内容

### 知识库健康标准

- **文件大小**：每个源文件 < 500 行，超过则按子主题拆分
- **出处标注**：每条知识至少标注积累自哪个项目，便于日后验证/更新
- **定期修剪**：过时的 workaround（如针对已修复的 OS bug）应标记删除
- **skill description 同步**：源文件增加新主题后，更新对应 skill 的 description 确保触发词覆盖

### Skill 架构原则

- **一个领域一个 skill**：避免碎片化，领域级 skill 作为唯一入口
- **薄壳模式**：SKILL.md 只维护源文件列表，知识内容在 AIKnowledge 中迭代
- **源文件列表即覆盖范围**：新增源文件时同步更新 SKILL.md 的文件列表
- **不再创建子 skill**：除非出现一个完全独立的新领域（非当前领域的子集）
